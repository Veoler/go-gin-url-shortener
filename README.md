# Сервер сокращателя ссылок

## Постановка задачи

Маркетинговое агентство ведёт рекламные кампании и хочет использовать короткие ссылки вместо длинных URL.
Фронтендеры уже сделали интерфейс, а твоя задача — написать backend-сервер на Go.

Сервер должен:

- хранить список коротких ссылок;
- позволять добавлять, изменять и удалять ссылки;
- по короткому `slug` перенаправлять пользователя на исходный URL.

## Техническое задание

### Сущность «ссылка»

Каждая ссылка должна содержать следующую информацию:

- `id` — уникальный идентификатор (целое число);
- `slug` — короткий человекочитаемый идентификатор (строка, используется в адресе, например `/r/course`);
- `url` — исходный полный адрес (строка, например `https://intocode.ru/courses/go/`).

Пример объекта ссылки:

```json
{
  "id": 1,
  "slug": "course",
  "url": "https://intocode.ru/courses/go/"
}
```

### Эндпоинты

В итоговой версии сервера должны быть реализованы следующие эндпоинты:

1. `GET /links` – список всех ссылок.

   Формат ответа:

   ```json
   [
     {
       "id": 1,
       "slug": "course",
       "url": "https://example.com/super-long-course-url"
     },
     {
       "id": 2,
       "slug": "blog",
       "url": "https://example.com/blog"
     }
   ]
   ```

2. `GET /links/:slug` – информация об одной ссылке по её `slug`.

   Пример запроса:

   ```
   GET /links/course
   ```

   Пример ответа:

   ```json
   {
     "id": 1,
     "slug": "course",
     "url": "https://example.com/super-long-course-url"
   }
   ```

3. `POST /links` – создание новой ссылки.

   Тело запроса:

   ```json
   {
     "slug": "course",
     "url": "https://example.com/super-long-course-url"
   }
   ```

   Пример ответа (созданная ссылка с присвоенным `id`):

   ```json
   {
     "id": 3,
     "slug": "course",
     "url": "https://example.com/super-long-course-url"
   }
   ```

4. `PATCH /links/:id` – изменение существующей ссылки по `id`.

   Пример запроса:

   ```
   PATCH /links/3
   ```

   Тело запроса (тот же формат, что и при создании):

   ```json
   {
     "slug": "course-new",
     "url": "https://example.com/updated-url"
   }
   ```

   Пример ответа:

   ```json
   {
     "id": 3,
     "slug": "course-new",
     "url": "https://example.com/updated-url"
   }
   ```

5. `DELETE /links/:id` – удаление ссылки по `id`.

   Пример запроса:

   ```
   DELETE /links/3
   ```

   В ответ можно вернуть, например:

   ```json
   { "status": "deleted" }
   ```

6. `DELETE /links/slug/:slug` – удаление ссылки по `slug`.

   Пример запроса:

   ```
   DELETE /links/slug/course
   ```

   Пример ответа:

   ```json
   { "status": "deleted" }
   ```

7. `GET /r/:slug` – редирект по короткой ссылке.

   Пример запроса:

   ```
   GET /r/course
   ```

   Сервер должен найти ссылку по `slug` и переадресовать клиента на полный URL.

   Информация для изучения: https://gin-gonic.com/ru/docs/examples/redirects/

### Обработка ошибок

Если операция выполняется неуспешно, сервер должен вернуть JSON с описанием ошибки:

```json
{ "error": "Описание ошибки" }
```

Примеры ситуаций:

- ссылка с таким `id` или `slug` не найдена:

  ```json
  { "error": "Ссылка не найдена" }
  ```

- при создании или обновлении ссылка с таким `slug` уже существует (эта проверка появится в отдельном релизе):

  ```json
  { "error": "Ссылка с таким slug уже существует" }
  ```

- передан некорректный URL (эта проверка тоже будет добавлена позже):

  ```json
  { "error": "Некорректный формат URL" }
  ```

Рекомендуется использовать подходящие HTTP-коды (`400`, `404`, `500` и т.д.), но главное — всегда возвращать объект с ключом `error` при ошибке.

## Releases

### Release 0: проектирование архитектуры

Подумай:

- какую структуру будет иметь сущность `Link` (поля и их типы);
- какие функции и сервисы будут нужны:

  - получить все ссылки;
  - найти по `id`;
  - найти по `slug`;
  - создать ссылку;
  - обновить ссылку;
  - удалить ссылку по `id` и по `slug`;

- как будут выглядеть роуты в `main.go` и какие обработчики к ним привязаны;

Когда в голове появится общая схема проекта — переходи к следующему релизу.

### Release 1: скелет сервера

1. Создай новый Go-проект.
2. Установи gin.
3. В файле `main.go`:

   - подключи необходимые пакеты;
   - создай `gin.Default`;
   - запусти сервер.

На этом этапе никаких реальных ссылок ещё не нужно — только живой сервер, который слушает порт.

### Release 2: структура ссылки

1. Опиши структуру `Link`, например:

   ```go
   type Link struct {
       ID   int    `json:"id"`
       Slug string `json:"slug"`
       URL  string `json:"url"`
   }
   ```

### Release 3: простое хранилище в памяти

На этом этапе мы сделаем простое хранилище ссылок в памяти на время работы сервера.

1. Создай слайс или мапу для хранения ссылок, например:

   ```go
   var links []Link
   ```

2. Подумай, какие вспомогательные функции тебе удобно завести:

   - получить все ссылки;
   - найти по `id`;
   - найти по `slug`;
   - добавить новую ссылку (с генерацией нового `id`);
   - обновить ссылку по `id`;
   - удалить по `id` и по `slug`.

3. Реализуй эти функции так, чтобы они работали **только с памятью** (слайсы/мапы) и не зависели от Gin. Хендлеры Gin будут просто вызывать их.

### Release 4: роуты для основных операций (CRUD-роуты)

Сначала реализуй первый эндпоинт для выдачи всех ссылок, которые сейчас есть в памяти:

- `GET /links` — возвращает весь список ссылок в формате:

  ```json
  [
    {
      "id": 1,
      "slug": "course",
      "url": "https://example.com/super-long-course-url"
    }
  ]
  ```

Претестируй в Postman. Если всё ок, то двигайся дальше.

- `POST /links` — создаёт новую ссылку:

  - принимает тело запроса:

    ```json
    {
      "slug": "course",
      "url": "https://example.com/super-long-course-url"
    }
    ```

  - присваивает новой ссылке уникальный `id` (например, `maxID + 1`);
  - добавляет её в слайс (или мапы);
  - возвращает созданный объект:

    ```json
    {
      "id": 1,
      "slug": "course",
      "url": "https://example.com/super-long-course-url"
    }
    ```

Также протестируй правильность работы в Postman. Только если первые два роута работают правильно (один выводит список ссылок, второй добавляет новую ссылку), то двигайся дальше.

`GET /links/:slug` – поиск ссылки по `slug`:

- если найдено — вернуть объект:

  ```json
  {
    "id": 1,
    "slug": "course",
    "url": "https://example.com/super-long-course-url"
  }
  ```

- если нет — вернуть:

  ```json
  { "error": "Ссылка не найдена" }
  ```

`PATCH /links/:id` – изменение ссылки по `id`:

- принимает тело запроса:

  ```json
  {
    "slug": "course-new",
    "url": "https://example.com/updated-url"
  }
  ```

– находит ссылку по `id` в памяти;
– меняет её поля и сохраняет изменённую структуру обратно в своё хранилище;
– возвращает обновлённый объект.

`DELETE /links/:id` – удаление ссылки по `id`.

`DELETE /links/slug/:slug` – удаление ссылки по `slug`.

### Release 5: редирект по короткой ссылке

Реализуй эндпоинт `GET /r/:slug`:

1. По `slug` найди нужную ссылку.
2. Если не найдена — верни:

   ```json
   { "error": "Ссылка не найдена" }
   ```

3. Если найдена — верни редирект:

   - статус `302` (или `301`, по желанию);
   - заголовок `Location` равен значению `url`.

Протестирвовать этот роут можно прямо в браузере. При переходе на ссылку вида `http://localhost:8080/r/[тут slug из списка]` тебя должно переадресовать на полный url, относящийся к этому slug.

### Release 6\*: проверка на дублирование `slug`

Добавь проверку уникальности `slug` при создании и обновлении ссылки.

Требования:

1. При `POST /links`, если в системе уже есть ссылка с таким `slug`, нужно вернуть ошибку:

   ```json
   { "error": "Ссылка с таким slug уже существует" }
   ```

   (и подходящий HTTP-код, например `400 Bad Request`).

2. При `PATCH /links/:id`, если новый `slug` уже используется другой ссылкой, также нужно вернуть такую же ошибку.

Сделай коммит.

### Release 7\*: проверка корректности формата URL

Теперь добавь валидацию поля `url`.

Требования:

1. При `POST /links` и `PATCH /links/:id` сервер должен проверять, что передан корректный URL.
2. Если URL некорректен, вернуть:

   ```json
   { "error": "Некорректный формат URL" }
   ```

   (и, например, статус `400 Bad Request`).

Подсказка по реализации:

- Посмотри стандартный пакет `net/url`.
- В нём есть функция `url.ParseRequestURI` или `url.Parse`, которые помогают проверить строку как URL.
- Пример подхода (идея, не обязательно копировать дословно):

  ```go
  import "net/url"

  func isValidURL(raw string) bool {
      parsed, err := url.ParseRequestURI(raw)
      if err != nil {
          return false
      }

      return true
  }
  ```

Вызови эту проверку перед сохранением ссылки.
Если URL не проходит проверку — верни ошибку.
